Splunk Enterprise — Install & Configure

Foundation · Host Machine or Dedicated VM
Required
1.1

Download & Install Splunk Enterprise (Linux Host)

▶

Install Splunk Enterprise on your host machine or a dedicated Linux VM. Splunk will act as both the indexer and search head for this lab. The free trial supports up to 500MB/day of indexed data — sufficient for a lab environment.

ℹ
Create a free Splunk account at splunk.com, then download Splunk Enterprise for Linux (.deb or .rpm). The following uses the .deb package for Ubuntu/Debian hosts.
bash 
# Download Splunk Enterprise (replace version with latest)
wget -O splunk-9.x.x-linux-amd64.deb \
  'https://download.splunk.com/products/splunk/releases/9.x.x/linux/splunk-9.x.x-linux-amd64.deb'

# Install the package
sudo dpkg -i splunk-9.x.x-linux-amd64.deb

# Start Splunk and accept license
sudo /opt/splunk/bin/splunk start --accept-license

# Enable Splunk to start on boot
sudo /opt/splunk/bin/splunk enable boot-start

# Set admin password (you'll be prompted)
sudo /opt/splunk/bin/splunk edit user admin -password YourSecurePass123! -role admin -auth admin:changeme
bash — verify install 
# Check Splunk status
sudo /opt/splunk/bin/splunk status

# Access the web UI from your browser:
http://<SPLUNK_HOST_IP>:8000
# Default credentials: admin / YourSecurePass123!
✓
VMware Tip: Note the IP address of your Splunk VM. In VMware, use ip addr show to find it. Set a static IP in VMware's virtual network settings to avoid IP changes on reboot.
1.2

Configure Splunk to Receive Data (Enable Listening Port)

▶

Enable Splunk's receiving port so Universal Forwarders can send data. Port 9997 is the standard forwarder receiving port. This must be done before connecting any forwarder.

bash — via CLI 
# Enable receiving on port 9997
sudo /opt/splunk/bin/splunk enable listen 9997 -auth admin:YourSecurePass123!

# Verify the port is listening
ss -tlnp | grep 9997

Alternatively, via the Splunk Web UI:

  • Navigate to Settings → Forwarding and receiving
  • Click Configure receiving → New Receiving Port
  • Enter port 9997 and save
  • Confirm the port shows as Enabled
⚠
Open port 9997 in your firewall: sudo ufw allow 9997/tcp on the Splunk host. For VMware, ensure the VM's virtual network allows inter-VM communication (use NAT or Host-Only with the same subnet).
1.3

Create Custom Indexes for Log Separation

▶

Create dedicated indexes to separate logs by source type. This improves search performance, access control, and makes dashboard queries cleaner.

bash — create indexes via CLI 
# Create Linux/Ubuntu log index
sudo /opt/splunk/bin/splunk add index linux_logs -auth admin:YourSecurePass123!

# Create Windows log index
sudo /opt/splunk/bin/splunk add index windows_logs -auth admin:YourSecurePass123!

# Create security events index (attack simulation data)
sudo /opt/splunk/bin/splunk add index soc_security -auth admin:YourSecurePass123!
linux_logs

Syslog, auth.log, kern.log, UFW firewall logs from Ubuntu

windows_logs

Windows Event Log, Security, System, Sysmon events from WinServer

02

Ubuntu VM — Universal Forwarder Setup

Endpoint Configuration · Ubuntu 20.04 / 22.04
Required
2.1

Install Splunk Universal Forwarder on Ubuntu

▶

The Universal Forwarder (UF) is a lightweight Splunk agent that collects and forwards logs to the Splunk indexer. It runs as a background service and has minimal system overhead.

bash — Ubuntu VM terminal 
# Download the Universal Forwarder
wget -O splunkforwarder-9.x.x-linux-amd64.deb \
  'https://download.splunk.com/products/universalforwarder/releases/9.x.x/linux/splunkforwarder-9.x.x-linux-amd64.deb'

# Install
sudo dpkg -i splunkforwarder-9.x.x-linux-amd64.deb

# Start and configure admin credentials
sudo /opt/splunkforwarder/bin/splunk start --accept-license \
  --answer-yes --no-prompt

# Enable auto-start on boot
sudo /opt/splunkforwarder/bin/splunk enable boot-start
2.2

Point the Forwarder to Splunk Indexer

▶

Configure the forwarder to send all collected data to your Splunk Enterprise indexer at port 9997. Replace SPLUNK_IP with the actual IP of your Splunk server.

bash 
# Add the Splunk indexer as a forwarding target
sudo /opt/splunkforwarder/bin/splunk add forward-server \
  SPLUNK_IP:9997 \
  -auth admin:YourSecurePass123!

# Verify the connection
sudo /opt/splunkforwarder/bin/splunk list forward-server \
  -auth admin:YourSecurePass123!
✓
You should see Active forwards: SPLUNK_IP:9997 in the output. If connection fails, verify firewall rules on both VMs and confirm port 9997 is open.
2.3

Configure Log Inputs (inputs.conf)

▶

Configure which log files the forwarder monitors. Create a custom inputs.conf that monitors the most important Linux security logs. This is the most critical configuration file for your Ubuntu endpoint.

bash — create inputs.conf 
sudo nano /opt/splunkforwarder/etc/system/local/inputs.conf
ini — inputs.conf content 
# Authentication log - SSH logins, sudo, failed auth
[monitor:///var/log/auth.log]
disabled = false
index = linux_logs
sourcetype = linux_secure
host_segment = 3

# Syslog - kernel messages, system events
[monitor:///var/log/syslog]
disabled = false
index = linux_logs
sourcetype = syslog

# UFW firewall logs - blocked/allowed connections
[monitor:///var/log/ufw.log]
disabled = false
index = linux_logs
sourcetype = linux_ufw

# Kernel log
[monitor:///var/log/kern.log]
disabled = false
index = linux_logs
sourcetype = linux_kern

# Apache/Nginx access logs (if web server installed)
[monitor:///var/log/apache2/access.log]
disabled = false
index = linux_logs
sourcetype = access_combined
bash — restart forwarder 
sudo /opt/splunkforwarder/bin/splunk restart
⚠
The forwarder runs as root to access restricted log files. Ensure the splunk user has read access: sudo chmod o+r /var/log/auth.log or run the forwarder as root.
2.4

Enable UFW Firewall & Auditd for Richer Logs

▶

Enable and configure UFW (firewall) and auditd (Linux audit daemon) to generate richer security telemetry that makes your Splunk dashboard more interesting.

bash — UFW setup 
# Enable UFW with logging
sudo ufw enable
sudo ufw logging on
sudo ufw allow 22/tcp   # Allow SSH
sudo ufw allow 80/tcp   # Allow HTTP (for attack simulation)
sudo ufw status verbose

# Install and configure auditd
sudo apt install auditd audispd-plugins -y
sudo systemctl enable auditd
sudo systemctl start auditd

# Add useful audit rules
sudo auditctl -w /etc/passwd -p wa -k passwd_changes
sudo auditctl -w /etc/shadow -p wa -k shadow_changes
sudo auditctl -w /bin/su -p x -k priv_esc
sudo auditctl -w /usr/bin/sudo -p x -k sudo_use

Windows Server — Universal Forwarder & Sysmon

Endpoint Configuration · Windows Server 2019/2022
Required
3.1

Install Sysmon for Enhanced Windows Telemetry

▶

Sysmon (System Monitor) dramatically improves Windows visibility by logging process creation, network connections, file operations, and more. Use the SwiftOnSecurity config — the gold standard for lab environments.

powershell — run as Administrator 
# Download Sysmon from Microsoft Sysinternals
Invoke-WebRequest -Uri "https://download.sysinternals.com/files/Sysmon.zip" -OutFile "C:\Tools\Sysmon.zip"
Expand-Archive -Path "C:\Tools\Sysmon.zip" -DestinationPath "C:\Tools\Sysmon"

# Download SwiftOnSecurity Sysmon config (recommended baseline)
Invoke-WebRequest -Uri "https://raw.githubusercontent.com/SwiftOnSecurity/sysmon-config/master/sysmonconfig-export.xml" -OutFile "C:\Tools\sysmonconfig.xml"

# Install Sysmon with the config
C:\Tools\Sysmon\Sysmon64.exe -accepteula -i C:\Tools\sysmonconfig.xml

# Verify Sysmon is running
Get-Service Sysmon64 | Select-Object Status, DisplayName
✓
Sysmon events appear in Windows Event Log → Applications and Services Logs → Microsoft → Windows → Sysmon. Event ID 1 = Process Create, Event ID 3 = Network Connection, Event ID 11 = File Create.
3.2

Install Splunk Universal Forwarder on Windows Server

▶

Download and install the Windows Universal Forwarder MSI. The installer allows you to configure the receiving indexer address directly during setup.

powershell — silent MSI install 
# Download the Windows UF installer
Invoke-WebRequest -Uri "https://download.splunk.com/products/universalforwarder/releases/9.x.x/windows/splunkforwarder-9.x.x-x64-release.msi" -OutFile "C:\Tools\splunkforwarder.msi"

# Silent installation with indexer pointing to Splunk server
msiexec.exe /i C:\Tools\splunkforwarder.msi RECEIVING_INDEXER="SPLUNK_IP:9997" AGREETOLICENSE=Yes SPLUNKUSERNAME=admin SPLUNKPASSWORD=YourSecurePass123! /quiet

# Add forward server (if not set during install)
& "C:\Program Files\SplunkUniversalForwarder\bin\splunk.exe" add forward-server SPLUNK_IP:9997 -auth admin:YourSecurePass123!

Configure Windows Event Log Inputs

▶

Create an inputs.conf to collect Windows Event Logs, Sysmon events, and PowerShell logs. This is the core of Windows endpoint visibility.

powershell — open inputs.conf

sourcetype = WinEventLog:Security
renderXml = false
evt_resolve_ad_obj = 1

# System Event Log
[WinEventLog://System]
disabled = false
index = windows_logs
sourcetype = WinEventLog:System

# Application Event Log
[WinEventLog://Application]
disabled = false
index = windows_logs
sourcetype = WinEventLog:Application

# Sysmon operational log
[WinEventLog://Microsoft-Windows-Sysmon/Operational]
disabled = false
index = windows_logs
sourcetype = XmlWinEventLog:Microsoft-Windows-Sysmon/Operational
renderXml = true

# PowerShell Operational log
[WinEventLog://Microsoft-Windows-PowerShell/Operational]
disabled = false
index = windows_logs

powershell — restart forwarder 
Restart-Service SplunkForwarder

ℹ
Key Windows Security Event IDs to know: 4624 (Logon), 4625 (Failed Logon), 4648 (Logon with explicit creds), 4720 (Account created), 4740 (Account locked out), 4688 (Process created).
3.4

Enable Windows Advanced Audit Policy

▶

Windows default audit policy generates minimal logs. Enable advanced auditing to capture the events that matter most for security monitoring.

powershell — enable audit policies 
# Enable comprehensive audit policies
auditpol /set /category:"Account Logon" /success:enable /failure:enable
auditpol /set /category:"Logon/Logoff" /success:enable /failure:enable
auditpol /set /category:"Account Management" /success:enable /failure:enable
auditpol /set /category:"Privilege Use" /success:enable /failure:enable
auditpol /set /category:"Detailed Tracking" /success:enable /failure:enable
auditpol /set /category:"Object Access" /success:enable /failure:enable
auditpol /set /category:"System" /success:enable /failure:enable

# Enable PowerShell Script Block Logging (critical for PS attacks)
reg add "HKLM\SOFTWARE\Policies\Microsoft\Windows\PowerShell\ScriptBlockLogging" /v EnableScriptBlockLogging /t REG_DWORD /d 1 /f

# Verify policies are set
auditpol /get /category:*

4.1

Core SPL Queries for Your Dashboard Panels

▶

These SPL (Splunk Processing Language) queries form the backbone of your SOC dashboard. Each query powers a specific panel visualizing a different aspect of your environment.

Panel 1 — Failed Login Attempts (Brute Force Detection)
spl 
index=linux_logs sourcetype=linux_secure 
  ("Failed password" OR "authentication failure" OR "Invalid user")
| rex "from (?P<src_ip>\d+\.\d+\.\d+\.\d+)"
| stats count as failures by src_ip
| sort -failures
| head 20
| eval severity=case(failures>=100,"CRITICAL",failures>=50,"HIGH",failures>=10,"MEDIUM",true(),"LOW")
Panel 2 — Windows Failed Logins by User
spl 
index=windows_logs sourcetype=WinEventLog:Security 
  EventCode=4625
| stats count as failed_logins by Account_Name, IpAddress
| where failed_logins > 3
| sort -failed_logins
| rename Account_Name as "Target User", IpAddress as "Source IP", failed_logins as "Failed Attempts"
Panel 3 — Real-time Event Count by Host (Timechart)
spl 
index=(linux_logs OR windows_logs)
| timechart count by host span=5m
| fillnull value=0
Panel 4 — Sysmon Network Connections (Outbound)
spl 
index=windows_logs sourcetype=XmlWinEventLog:Microsoft-Windows-Sysmon/Operational 
  EventCode=3
| spath
| stats count by Image, DestinationIp, DestinationPort
| sort -count
| head 25
Panel 5 — UFW Firewall Block Events (Geo)
spl 
index=linux_logs sourcetype=linux_ufw "[UFW BLOCK]"
| rex "SRC=(?P<src_ip>\d+\.\d+\.\d+\.\d+) DST=(?P<dst_ip>\d+\.\d+\.\d+\.\d+).*DPT=(?P<dst_port>\d+)"
| iplocation src_ip
| stats count as blocked by src_ip, Country, dst_port
| sort -blocked
Panel 6 — New Processes Created (Sysmon EventID 1)
spl 
index=windows_logs sourcetype=XmlWinEventLog:Microsoft-Windows-Sysmon/Operational 
  EventCode=1
| spath
| eval suspicious=if(match(CommandLine,"(powershell.*-enc|cmd.*\/c|wscript|cscript|mshta|regsvr32|rundll32)"),"YES","NO")
| table _time, Image, CommandLine, User, ParentImage, suspicious
| sort -_time

6.1

Configure Splunk Alerts for Key Security Events

▶

Splunk alerts fire when a search returns results matching your conditions. Here are the most important alerts for your SOC lab.

Alert Name Trigger Condition Severity Schedule
Brute Force SSH >10 failed logins from same IP in 5 min Critical Every 5 min
Windows Account Lockout EventCode=4740 any occurrence High Real-time
Suspicious PowerShell CommandLine contains -EncodedCommand or IEX High Every 15 min
New Local User Created EventCode=4720 or "new user" in auth.log Medium Every 1 hr
UFW Spike (Port Scan) >100 UFW BLOCK events from same IP in 1 min High Every 1 min
Sysmon Network to Rare Port EventCode=3 DestinationPort IN (4444,1337,31337) Critical Real-time
spl — brute force alert query 
# Save this as an alert: "Brute Force SSH Detection"
index=linux_logs sourcetype=linux_secure "Failed password"
| rex "from (?P<src_ip>\d+\.\d+\.\d+\.\d+)"
| bucket _time span=5m
| stats count by src_ip, _time
| where count > 10

To create an alert in Splunk UI:

  • Run the query in Search & Reporting
  • Click Save As → Alert
  • Set Alert Type: Scheduled (cron) or Real-time
  • Set Trigger conditions: Number of results is greater than 0
  • Add Alert Action: Add to Triggered Alerts (visible in SOC dashboard)
  • Set Severity level for the triggered alert
6.2

Advanced Correlation Searches

▶

Correlation searches connect events across multiple sources to detect complex attack patterns. These are the hallmark of a mature SOC.

spl — lateral movement detection 
# Detect: failed Linux SSH logins followed by Windows activity from same IP
index=linux_logs "Failed password"
| rex "from (?P<src_ip>\d+\.\d+\.\d+\.\d+)"
| stats count as linux_failures by src_ip
| join src_ip [
    search index=windows_logs EventCode=4624
    | rename IpAddress as src_ip
    | stats count as win_logins by src_ip
]
| where linux_failures > 5 AND win_logins > 0
| eval alert="Possible lateral movement from: ".src_ip
spl — impossible travel detection 
# Detect same user logging in from different countries within 1 hour
index=windows_logs EventCode=4624 Account_Name!="SYSTEM" Account_Name!="ANONYMOUS LOGON"
| iplocation IpAddress
| stats dc(Country) as countries values(Country) as country_list by Account_Name
| where countries > 1
| eval alert="Impossible travel: ".Account_Name." from: ".mvjoin(country_list,", ")

Network Change Recovery — Home ↔ College WiFi

IP Reconfiguration · Portability · One-Command Fix
Must Do on Network Change
Why This Breaks
When you built this at home, your VMware VMs were assigned IPs by your home router's DHCP — something like 192.168.1.x. You hardcoded those IPs into every forwarder config.
At college, a different DHCP server hands out new IPs — likely 10.x.x.x or 172.16.x.x. The old IP in every config file is now wrong. Forwarders silently fail.
The fix is simple: find the new Splunk server IP and update that one address in two config files — one on Ubuntu, one on Windows. Take 3 minutes max.

Step 1 — Find All New IP Addresses After Connecting to New WiFi

Step 2 — Update Ubuntu Forwarder to Point at New Splunk IP

        ▶
      

      

        

          The Ubuntu forwarder stores the Splunk indexer's IP in outputs.conf. 
          You need to remove the old entry and add the new one. Run all commands on the Ubuntu VM.
        


        

          
bash — method A: CLI commands (quickest)

          
# 1. Remove the OLD Splunk server IP
sudo /opt/splunkforwarder/bin/splunk remove forward-server \
  OLD_SPLUNK_IP:9997 \
  -auth admin:YourSecurePass123!

# 2. Add the NEW Splunk server IP
sudo /opt/splunkforwarder/bin/splunk add forward-server \
  NEW_SPLUNK_IP:9997 \
  -auth admin:YourSecurePass123!

# 3. Restart the forwarder to apply
sudo /opt/splunkforwarder/bin/splunk restart

# 4. Verify the new connection is active
sudo /opt/splunkforwarder/bin/splunk list forward-server \
  -auth admin:YourSecurePass123!
# You should see: Active forwards: NEW_SPLUNK_IP:9997

        


        

          
bash — method B: edit outputs.conf directly

          
# View the current outputs.conf to see old IP
cat /opt/splunkforwarder/etc/system/local/outputs.conf

# The file looks like:
[tcpout]
defaultGroup = default-autolb-group

[tcpout:default-autolb-group]
server = OLD_SPLUNK_IP:9997   # ← change this line

# Use sed to replace old IP with new IP in one command:
sudo sed -i 's/OLD_SPLUNK_IP/NEW_SPLUNK_IP/g' \
  /opt/splunkforwarder/etc/system/local/outputs.conf

# Confirm the change:
grep server /opt/splunkforwarder/etc/system/local/outputs.conf

# Restart forwarder:
sudo /opt/splunkforwarder/bin/splunk restart

        


        

          ✓
          
Method A (CLI) is recommended because Splunk manages the config file cleanly. Method B (direct edit) is faster if you know the exact old IP and want a one-liner. Both work identically.

        ▶
      

      

        

          After updating both forwarders, open the Splunk Web UI and confirm events are arriving from both endpoints.

# An error shows:

              Symptom
              Likely Cause
              Fix
            

          
          
            

              No events in linux_logs
              Ubuntu outputs.conf still has old IP
              Re-run Step 7.2, verify with list forward-server
            

            

              No events in windows_logs

        7.5
        
Permanent Fix — Use VMware Host-Only Networking (Recommended)

        ▶
      

      

        

          This is the real solution that eliminates the problem completely. By switching all your SOC VMs to 
          Host-Only networking, they communicate over a private virtual network inside VMware 
          that never changes — regardless of whether you're on home WiFi, college WiFi, or no WiFi at all.
          You only set this up once.
        


        

          ℹ
          
How it works: VMware creates a virtual network adapter (VMnet1 by default for Host-Only) that lives entirely inside your laptop. VMs on this network get IPs like 192.168.56.x permanently — they never change because no external DHCP server is involved. The Splunk UI on port 8000 is still accessible from your laptop's browser.

        


        

          
vmware workstation — step-by-step

          
═══════════════════════════════════════════════════
  Configure VMware Host-Only Network (do this once)
═══════════════════════════════════════════════════

STEP A — Open VMware Workstation
  → Edit → Virtual Network Editor → Run as Administrator

STEP B — Select VMnet1 (Host-only)
  → Check: "Use local DHCP service to distribute IP addresses"
  → Subnet IP: 192.168.56.0
  → Subnet Mask: 255.255.255.0
  → Click Apply

STEP C — For EACH VM (Splunk host, Ubuntu VM, Windows Server):
  → Right-click VM → Settings → Network Adapter
  → Change from "NAT" to "Host-only: VMnet1"
  → Click OK

STEP D — Boot all VMs and check their new stable IPs
  Ubuntu/Splunk: ip addr show  → expect 192.168.56.x
  Windows:       ipconfig        → expect 192.168.56.x

        


        

          
bash — assign static IPs (prevents DHCP reassignment)

          
# On Ubuntu: assign a static IP using netplan
sudo nano /etc/netplan/00-installer-config.yaml

# Replace contents with:
network:
  version: 2
  ethernets:
    ens33:           # replace ens33 with your interface name
      addresses: [192.168.56.10/24]   # Splunk host gets .10
      dhcp4: no
      # No gateway needed — host-only has no internet access

sudo netplan apply

═══════════════════════════
  Recommended static IP plan:
  Splunk Server  → 192.168.56.10
  Ubuntu VM      → 192.168.56.20
  Windows Server → 192.168.56.30
═══════════════════════════

        


        

          
powershell — set static IP on Windows Server

          
# Find the adapter name first
Get-NetAdapter | Select-Object Name, InterfaceDescription

# Set static IP (replace "Ethernet0" with your adapter name)
New-NetIPAddress -InterfaceAlias "Ethernet0" -IPAddress 192.168.56.30 -PrefixLength 24

# Verify
Get-NetIPAddress -InterfaceAlias "Ethernet0"

        


        

          ✓
          

            After this setup, your forwarder configs never need to change again. Splunk server is always 
            192.168.56.10:9997. Plug into home WiFi, college WiFi, or no WiFi — the SOC keeps running. 
            Access Splunk UI at http://192.168.56.10:8000 from your laptop browser every time.
          

        

      

    


    
    

      

        7.6
        
Bonus — One-Command IP Update Script for Quick Fixes

        ▶
      

      

        

          If you still want to use NAT/bridged networking and just want to fix IPs faster, 
          save these scripts on each VM. Run them whenever you switch networks.
        


        

          
bash — save as ~/fix-splunk-ip.sh on Ubuntu VM

          
#!/bin/bash
# fix-splunk-ip.sh — run this every time you change networks

echo "Enter the NEW Splunk server IP:"
read NEW_IP

# Validate IP format
if ! [[ $NEW_IP =~ ^[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+$ ]]; then
  echo "Invalid IP format. Exiting."; exit 1
fi

# Get old IP from current config
OLD_IP=$(grep "^server" /opt/splunkforwarder/etc/system/local/outputs.conf | awk -F'[= :]' '{print $2}' | tr -d ' ')
echo "Replacing old IP: $OLD_IP → $NEW_IP"

# Update outputs.conf
sudo sed -i "s/${OLD_IP}/${NEW_IP}/g" /opt/splunkforwarder/etc/system/local/outputs.conf

# Restart forwarder
sudo /opt/splunkforwarder/bin/splunk restart --answer-yes

echo "✓ Done. Forwarder now pointing to $NEW_IP:9997"
echo "Run: sudo /opt/splunkforwarder/bin/splunk list forward-server -auth admin:YourPass to verify"

        


        

          
bash — make script executable

          
chmod +x ~/fix-splunk-ip.sh
# Run it next time with:
sudo ~/fix-splunk-ip.sh

        


        

          
powershell — save as C:\fix-splunk-ip.ps1 on Windows

          
# fix-splunk-ip.ps1 — run as Administrator when network changes

$NEW_IP = Read-Host "Enter the NEW Splunk server IP"
$conf   = "C:\Program Files\SplunkUniversalForwarder\etc\system\local\outputs.conf"

# Show current config
Write-Host "Current config:" -ForegroundColor Cyan
Get-Content $conf

# Replace IP in config file
$content = Get-Content $conf
$OLD_IP  = ($content | Select-String "server\s*=").ToString() -replace '.*=\s*(\d+\.\d+\.\d+\.\d+).*','$1'
Write-Host "Replacing $OLD_IP → $NEW_IP" -ForegroundColor Yellow

$content -replace [regex]::Escape($OLD_IP), $NEW_IP | Set-Content $conf

# Restart forwarder
Restart-Service SplunkForwarder
Write-Host "✓ Done. Forwarder restarted pointing to $($NEW_IP):9997" -ForegroundColor Green

        


        

          ✓
          
Save the PowerShell script and right-click it → Run with PowerShell (as Admin). It'll prompt you for the new IP, update the config, and restart the service — the whole fix in under 30 seconds.
