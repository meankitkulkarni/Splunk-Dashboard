Clean Slate — Remove Any Previous Splunk Installation

Do This First · On All Three Machines · Before Installing Anything New
Do First
🧹
Why this phase exists: If a previous Splunk install failed, was partially set up, or left behind broken files or services, a fresh install on top of it will inherit those problems. This phase wipes everything cleanly so you truly start from zero. Run this on all three machines before Phase 1.
⚠
"Command not found" on Ubuntu is completely normal. If you see sudo: /opt/splunkforwarder/bin/splunk: command not found (as in the screenshot), it simply means the Splunk binary was never placed there — either it was never installed, or a previous attempt failed before it could finish. The folder may not exist at all. Do not worry about this error — the cleanup steps below are written to handle it gracefully. Just move to the next command.
0.1

🪟 Windows Host — Remove Splunk Enterprise

►

Remove Splunk Enterprise from your Windows PC completely — the service, the install folder, and the firewall rules. Do this even if a previous install seemed to work partially.

📋 Instructions — open PowerShell as Administrator on your Windows PC
  1. Run Step A to stop the service and find the uninstall ID (the long GUID string)
  2. Then go to Settings → Apps → search "Splunk" → Uninstall (easiest), OR use the GUID with the msiexec command in Step B
  3. After uninstall, run Step C to delete leftover files and firewall rules
powershell — Step A: Stop service & find uninstall GUID Windows Host 
# Stop Splunk if it is running (safe to run even if Splunk is not installed)
Stop-Service SplunkD -Force -ErrorAction SilentlyContinue

# Find the Splunk uninstall GUID — note the IdentifyingNumber value
Get-WmiObject -Class Win32_Product | Where-Object { $_.Name -like "*Splunk*" } | Select-Object Name, IdentifyingNumber
# If this returns nothing: Splunk was not registered via MSI — skip Step B
powershell — Step B: Uninstall using the GUID from Step A Windows Host 
# Replace {PASTE-YOUR-GUID-HERE} with the IdentifyingNumber from Step A
# Example: msiexec.exe /x "{4A67D191-0000-0000-0000-000000000000}" /quiet
msiexec.exe /x "{PASTE-YOUR-GUID-HERE}" /quiet
# Wait ~30 seconds for the uninstall to complete before running Step C
powershell — Step C: Delete leftover files and firewall rules Windows Host 
# Delete Splunk install folder (safe even if it does not exist)
Remove-Item "C:\Program Files\Splunk" -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item "C:\Program Files\SplunkUniversalForwarder" -Recurse -Force -ErrorAction SilentlyContinue

# Remove firewall rules if they were added previously
Remove-NetFirewallRule -DisplayName "Splunk Forwarder Port" -ErrorAction SilentlyContinue
Remove-NetFirewallRule -DisplayName "Splunk Web UI" -ErrorAction SilentlyContinue

# Verify Splunk is fully gone — expect no output and "False"
Get-Service SplunkD -ErrorAction SilentlyContinue
Test-Path "C:\Program Files\Splunk"
# Good result: nothing from Get-Service, and False from Test-Path
0.2

🐧 Ubuntu VM — Remove Splunk Universal Forwarder

►

Each command below is independently safe to run. If you see "command not found", "No such file", or "package not installed" on any line — that part was already clean. Just run the next command.

📸
Regarding the error in your screenshot:
You ran sudo /opt/splunkforwarder/bin/splunk stop and got command not found.

This means the Splunk binary does not exist at that path — the forwarder was either never fully installed, or a previous attempt left nothing behind. This is not a problem. Simply run the three steps below. Step 1 uses a safe check before calling any Splunk command, so it will not throw that error again.
📋 Instructions — run inside the Ubuntu VM Terminal
  1. Open the Ubuntu VM → open a Terminal
  2. Run Steps 1, 2, 3, and 4 below one at a time
  3. Any "not found" or "already clean" message is expected and fine — keep going
bash — Step 1: Safely stop the forwarder if it exists Ubuntu VM 
# Checks if the binary exists BEFORE trying to stop it — no "command not found" error
[ -f /opt/splunkforwarder/bin/splunk ] &&   sudo /opt/splunkforwarder/bin/splunk stop ||   echo "Splunk binary not found — skipping stop (this is fine)"

[ -f /opt/splunkforwarder/bin/splunk ] &&   sudo /opt/splunkforwarder/bin/splunk disable boot-start ||   echo "Nothing to disable — skipping"
bash — Step 2: Delete the install folder Ubuntu VM 
# Force-delete the folder — safe even if it does not exist
sudo rm -rf /opt/splunkforwarder
echo "Folder deleted (or was already gone)"
bash — Step 3: Remove the package record and systemd service Ubuntu VM 
# Purge the dpkg record (safe even if never registered)
sudo dpkg --purge splunkforwarder 2>/dev/null || echo "Not in dpkg — already clean"

# Remove systemd service if it was registered
sudo systemctl stop SplunkForwarder 2>/dev/null || true
sudo systemctl disable SplunkForwarder 2>/dev/null || true
sudo rm -f /etc/systemd/system/SplunkForwarder.service
sudo systemctl daemon-reload
bash — Step 4: Verify it is gone Ubuntu VM 
# Check folder is gone
ls /opt/splunkforwarder 2>/dev/null || echo "Clean — /opt/splunkforwarder does not exist"

# Check dpkg has no Splunk record
dpkg -l | grep -i splunk || echo "Clean — no Splunk packages in dpkg"
# Good: both lines should echo the clean message

Splunk Enterprise — Install on Windows Host

Your Physical Windows 10/11 Machine · One-time Setup
Required
1.1

Download & Install Splunk Enterprise on Windows

▶

Splunk Enterprise is the core SIEM — it receives all logs, lets you search them, and runs your dashboards. You'll install it on your own Windows machine (the host), not inside any VM.

📋 Instructions — follow these steps first
  1. Go to splunk.com → create a free account → go to Free Trials & Downloads
  2. Download Splunk Enterprise for Windows — the .msi installer file
  3. Run the MSI installer → tick "Accept License" → choose Local System account
  4. When prompted, set your admin username and a strong password (remember this!)
  5. Finish the install — Splunk will start automatically as a Windows service
powershell Windows Host 
# Run these in PowerShell as Administrator after MSI install

# Check if Splunk service is running
Get-Service SplunkD | Select-Object Status, DisplayName

# If the service is stopped, start it
Start-Service SplunkD

# Find your Windows host IP address (note this — VMs will send logs here)
Get-NetIPAddress -AddressFamily IPv4 | Where-Object {$_.IPAddress -notlike "127.*" -and $_.IPAddress -notlike "169.*"} | Select-Object IPAddress, InterfaceAlias
✓
Once installed, open your browser and go to http://localhost:8000 — you should see the Splunk login page. Log in with the admin credentials you set during install.
1.2

Open Firewall Ports on Windows Host

▶

Your VMs need to reach Splunk on port 9997 to send logs. Your browser needs port 8000 for the Splunk UI. Windows Firewall blocks these by default — run these commands to open them.

📋 Instructions
  1. Open PowerShell as Administrator (right-click PowerShell → "Run as Administrator")
  2. Run the commands below to add firewall rules
  3. After running, verify the rules appear in Windows Defender Firewall → Inbound Rules
powershell — run as Administrator Windows Host 
# Allow VMs to send logs to Splunk (forwarder port)
New-NetFirewallRule -DisplayName "Splunk Forwarder Port" -Direction Inbound -Protocol TCP -LocalPort 9997 -Action Allow

# Allow browser access to Splunk Web UI
New-NetFirewallRule -DisplayName "Splunk Web UI" -Direction Inbound -Protocol TCP -LocalPort 8000 -Action Allow

# Verify port 9997 is now listening (Splunk must be running)
netstat -ano | findstr ":9997"
# Expected output: ... 0.0.0.0:9997 ... LISTENING

# Check firewall rules were added
Get-NetFirewallRule | Where-Object {$_.DisplayName -like "Splunk*"} | Select-Object DisplayName, Enabled, Direction
1.3

Enable Receiving Port & Create Log Indexes

▶

Tell Splunk to listen for incoming log data on port 9997, then create separate indexes (log buckets) for your Ubuntu logs and Windows Server logs. This keeps everything organized and makes dashboard queries faster.

📋 Instructions — do this in the Splunk Web UI
  1. Open http://localhost:8000 and log in
  2. Go to Settings → Forwarding and Receiving
  3. Click Configure Receiving → New Receiving Port
  4. Type 9997 and click Save — the port should show as Enabled
powershell — OR use the CLI instead Windows Host 
# Enable Splunk to receive data on port 9997
& "C:\Program Files\Splunk\bin\splunk.exe" enable listen 9997 -auth admin:YourSecurePass123!

# Create index for Ubuntu / Linux logs
& "C:\Program Files\Splunk\bin\splunk.exe" add index linux_logs -auth admin:YourSecurePass123!

# Create index for Windows Server VM logs
& "C:\Program Files\Splunk\bin\splunk.exe" add index windows_logs -auth admin:YourSecurePass123!

# Create index for security/attack event data
& "C:\Program Files\Splunk\bin\splunk.exe" add index soc_security -auth admin:YourSecurePass123!

# Confirm the indexes were created
& "C:\Program Files\Splunk\bin\splunk.exe" list index -auth admin:YourSecurePass123!
linux_logs
Auth logs, syslog, UFW firewall, kernel logs — all from the Ubuntu VM
windows_logs
Security events, Sysmon, PowerShell logs — all from the Windows Server VM

Ubuntu VM — Universal Forwarder Setup

VMware Guest · Ubuntu 20.04 / 22.04 · All commands run inside the VM
Required
ℹ

2.1

Install the Splunk Universal Forwarder on Ubuntu

▶

The Universal Forwarder (UF) is a lightweight Splunk agent. It collects log files and sends them to Splunk Enterprise on your Windows host. It uses almost no CPU or RAM.

📋 Instructions
  1. On your Windows PC, go to splunk.com → Free Trials & Downloads → Universal Forwarder
  2. Copy the download link for Linux .deb 64-bit
  3. Inside the Ubuntu VM terminal, use wget with that link to download it, then install with dpkg
bash — Ubuntu VM Terminal Ubuntu VM 
# Download the Universal Forwarder — replace the URL with the one from splunk.com
wget -O splunkforwarder.deb \
  'https://download.splunk.com/products/universalforwarder/releases/9.x.x/linux/splunkforwarder-9.x.x-linux-amd64.deb'

# Install it
sudo dpkg -i splunkforwarder.deb

# Start the forwarder for the first time and accept the license
sudo /opt/splunkforwarder/bin/splunk start --accept-license --answer-yes --no-prompt

# Make the forwarder start automatically when the VM boots
sudo /opt/splunkforwarder/bin/splunk enable boot-start

# Check it is running
sudo /opt/splunkforwarder/bin/splunk status
2.2

Point the Forwarder to Your Windows Host (Splunk Server)

▶

Tell the Ubuntu forwarder where to send its logs — your Windows host's IP address on port 9997. Replace WINDOWS_HOST_IP with the actual IP you found in Phase 1 (Step 1.1).

📋 How to find your Windows host IP (if you haven't already)
  1. On your Windows PC, open PowerShell and run: Get-NetIPAddress -AddressFamily IPv4
  2. Look for the IP on your network adapter (usually starts with 192.168.x.x or 10.x.x.x)
  3. That IP is what you put below as WINDOWS_HOST_IP
bash — Ubuntu VM Terminal Ubuntu VM 
# Tell the forwarder to send logs to your Windows host
sudo /opt/splunkforwarder/bin/splunk add forward-server \
  WINDOWS_HOST_IP:9997 \
  -auth admin:YourSecurePass123!

# Confirm the connection target is set correctly
sudo /opt/splunkforwarder/bin/splunk list forward-server \
  -auth admin:YourSecurePass123!
# You should see: Active forwards: WINDOWS_HOST_IP:9997
✓
If the connection fails, try ping WINDOWS_HOST_IP from the Ubuntu VM. If ping fails, check VMware network settings — both VMs and host must be on the same virtual network (use NAT or Host-Only mode).
2.3

Configure Which Log Files to Forward (inputs.conf)

▶

The inputs.conf file tells the forwarder which log files to monitor and send to Splunk. You'll create this file with the most useful Linux security logs.

📋 Instructions
  1. Run the first command below to open the file in nano (text editor)
  2. Type or paste the entire inputs.conf content shown in the second box
  3. Save and exit: press Ctrl+X, then Y, then Enter
  4. Restart the forwarder so it picks up the new config
bash — Step 1: Open the file Ubuntu VM 
sudo nano /opt/splunkforwarder/etc/system/local/inputs.conf
ini — Step 2: Paste this into the file Ubuntu VM 
# SSH logins, sudo activity, authentication failures
[monitor:///var/log/auth.log]
disabled = false
index = linux_logs
sourcetype = linux_secure

# General system messages and kernel events
[monitor:///var/log/syslog]
disabled = false
index = linux_logs
sourcetype = syslog

# UFW firewall — blocked and allowed connection attempts
[monitor:///var/log/ufw.log]
disabled = false
index = linux_logs
sourcetype = linux_ufw

# Kernel log messages
[monitor:///var/log/kern.log]
disabled = false
index = linux_logs
sourcetype = linux_kern

# Apache web server access log (for web attack simulation)
[monitor:///var/log/apache2/access.log]
disabled = false
index = linux_logs
sourcetype = access_combined
bash — Step 3: Restart the forwarder Ubuntu VM 
sudo /opt/splunkforwarder/bin/splunk restart

# Confirm the forwarder is sending data (look for "Connected to idx=" in the log)
sudo tail -20 /opt/splunkforwarder/var/log/splunk/splunkd.log | grep -i "connect\|forward\|error"
⚠
The forwarder needs read access to log files. If you get "permission denied" errors, run: sudo chmod o+r /var/log/auth.log. Alternatively, run the forwarder as root by adding SPLUNK_OS_USER=root to /opt/splunkforwarder/etc/splunk-launch.conf.
2.4

Enable UFW Firewall & Auditd for Better Logs

▶

By default, Ubuntu's firewall (UFW) is off and audit logging is minimal. Enable both so your attacks generate rich, visible security events in Splunk.

bash — Ubuntu VM Terminal Ubuntu VM 
# Enable UFW and turn on logging so blocked packets appear in ufw.log
sudo ufw enable
sudo ufw logging on
sudo ufw allow 22/tcp    # keep SSH open
sudo ufw allow 80/tcp    # keep HTTP open for attack simulation
sudo ufw status verbose

# Install auditd — generates detailed audit events for file/command access
sudo apt install auditd audispd-plugins -y
sudo systemctl enable auditd
sudo systemctl start auditd

# Add audit rules — these trigger alerts when sensitive files or commands are touched
sudo auditctl -w /etc/passwd -p wa -k passwd_changes
sudo auditctl -w /etc/shadow -p wa -k shadow_changes
sudo auditctl -w /bin/su -p x -k priv_esc
sudo auditctl -w /usr/bin/sudo -p x -k sudo_use

# Install Apache (web server) for web attack simulation later
sudo apt install apache2 -y
sudo systemctl start apache2
sudo systemctl enable apache2

3.1

Install Sysmon for Rich Windows Telemetry

▶

Sysmon (System Monitor) is a free Microsoft tool that logs detailed events that Windows normally misses — things like which process opened a network connection, or what command line was used to start a program. It makes your Windows endpoint much more visible in Splunk.

📋 Instructions
  1. Inside the Windows Server VM, open PowerShell as Administrator
  2. First create a folder for your tools: mkdir C:\Tools
  3. Run the commands below to download Sysmon and the recommended config file
  4. The last command installs Sysmon with that config — it starts running immediately
powershell — run as Administrator Windows Server VM 
# Create a tools folder
mkdir C:\Tools -Force

# Download Sysmon from Microsoft Sysinternals
Invoke-WebRequest -Uri "https://download.sysinternals.com/files/Sysmon.zip" -OutFile "C:\Tools\Sysmon.zip"
Expand-Archive -Path "C:\Tools\Sysmon.zip" -DestinationPath "C:\Tools\Sysmon" -Force

# Download the SwiftOnSecurity config (community-recommended baseline)
Invoke-WebRequest -Uri "https://raw.githubusercontent.com/SwiftOnSecurity/sysmon-config/master/sysmonconfig-export.xml" -OutFile "C:\Tools\sysmonconfig.xml"

# Install Sysmon using the config file
C:\Tools\Sysmon\Sysmon64.exe -accepteula -i C:\Tools\sysmonconfig.xml

# Confirm Sysmon is running
Get-Service Sysmon64 | Select-Object Status, DisplayName
✓
Sysmon logs appear in Event Viewer → Applications and Services Logs → Microsoft → Windows → Sysmon → Operational. Key Event IDs: 1 = Process Created, 3 = Network Connection, 11 = File Created, 13 = Registry Value Set.
3.2

▶

Install the Windows version of the Splunk Universal Forwarder. This sends Windows Event Logs, Sysmon events, and PowerShell logs to your Splunk Enterprise on the Windows host.

Configure Windows Event Log Inputs (inputs.conf)

▶

Create the inputs.conf file to tell the forwarder which Windows logs to collect. This file enables collection of Security events, Sysmon events, PowerShell logs, and more.

📋 Instructions
  1. Run the first command to open the inputs.conf file in Notepad
  2. Paste the entire config block (second box) into Notepad
  3. Save the file (Ctrl+S) and close Notepad
  4. Restart the forwarder service using the third command

# Security events — logins, failures, account changes (most important)
[WinEventLog://Security]
disabled = false
index = windows_logs
sourcetype = WinEventLog:Security
renderXml = false
evt_resolve_ad_obj = 1

# System events — service crashes, driver issues
[WinEventLog://System]
disabled = false
index = windows_logs
sourcetype = WinEventLog:System

# Application events
[WinEventLog://Application]
disabled = false
index = windows_logs
sourcetype = WinEventLog:Application

# Sysmon — rich process, network, file activity
[WinEventLog://Microsoft-Windows-Sysmon/Operational]
disabled = false
index = windows_logs
sourcetype = XmlWinEventLog:Microsoft-Windows-Sysmon/Operational
renderXml = true

# PowerShell script execution logs — catches encoded/suspicious PS commands
[WinEventLog://Microsoft-Windows-PowerShell/Operational]
disabled = false
index = windows_logs
sourcetype = WinEventLog:PowerShell

# DNS client queries — shows what domains this machine looks up
[WinEventLog://Microsoft-Windows-DNS-Client/Operational]
disabled = false
index = windows_logs
sourcetype = WinEventLog:DNS-Client
powershell — Step 3: Restart the forwarder Windows Server VM 
Restart-Service SplunkForwarder

# Confirm it's running
Get-Service SplunkForwarder | Select-Object Status, DisplayName

3.4

Enable Advanced Audit Policy & PowerShell Logging

▶

Windows default audit settings log very little. Run these commands on the Windows Server VM to turn on detailed logging so your attack simulations generate visible events in Splunk.

powershell — run as Administrator Windows Server VM 
# Enable all major audit categories
auditpol /set /category:"Account Logon" /success:enable /failure:enable
auditpol /set /category:"Logon/Logoff" /success:enable /failure:enable
auditpol /set /category:"Account Management" /success:enable /failure:enable
auditpol /set /category:"Privilege Use" /success:enable /failure:enable
auditpol /set /category:"Detailed Tracking" /success:enable /failure:enable
auditpol /set /category:"Object Access" /success:enable /failure:enable
auditpol /set /category:"System" /success:enable /failure:enable

# Enable PowerShell Script Block Logging (catches obfuscated/encoded PS commands)

# Confirm all audit policies are active
auditpol /get /category:*

Splunk Dashboard — SOC Command Center

Required
ℹ
Where to do this: Everything in this phase is done in your browser on your Windows host — open http://localhost:8000 and log into Splunk. You'll run SPL queries and build the dashboard there.
4.1

Verify Data is Arriving from Both VMs

▶

Before building the dashboard, confirm that Splunk is receiving logs from both VMs. Run this SPL query in Splunk's Search bar — you should see events from both your Ubuntu and Windows Server.

📋 How to run an SPL query
  1. In Splunk, click Search & Reporting in the top menu
  2. Paste the query into the big search bar at the top
  3. Set the time range to Last 60 minutes (top right)
  4. Click the green magnifying glass or press Enter
spl — data verification query Splunk Web UI 
index=(linux_logs OR windows_logs) earliest=-60m
| stats count as event_count, latest(_time) as last_seen by host, index, sourcetype
| eval last_seen=strftime(last_seen, "%H:%M:%S")
| sort -last_seen
✓
You should see rows for your Ubuntu VM (host column shows its hostname) and your Windows Server VM. If no data appears, go back to Phase 2 or 3 and check the forwarder is running and pointing to the right IP.
4.2

Core Dashboard SPL Queries (10 Panels)

▶

These 10 SPL queries form your complete SOC dashboard. Each one becomes a panel showing a different security view. Test each one in the Search bar first, then add it to the dashboard.

Panel 1 — SSH Brute Force Detection (Linux)
spl
index=linux_logs sourcetype=linux_secure
  ("Failed password" OR "authentication failure" OR "Invalid user")
| rex "from (?P<src_ip>\d+\.\d+\.\d+\.\d+)"
| stats count as failures by src_ip
| sort -failures
| head 20
| eval severity=case(failures>=100,"CRITICAL",failures>=50,"HIGH",failures>=10,"MEDIUM",true(),"LOW")
Panel 2 — Windows Failed Logins (Event ID 4625)
spl
index=windows_logs sourcetype=WinEventLog:Security EventCode=4625
| stats count as failed_logins by Account_Name, IpAddress
| where failed_logins > 3
| sort -failed_logins
